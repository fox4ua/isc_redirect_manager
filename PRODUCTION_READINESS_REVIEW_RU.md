# Анализ готовности модуля `isc_redirect_manager` к production

Дата: 2026-03-15
Формат анализа: статический аудит кода + базовые проверки окружения.

## Итоговый вердикт

**Статус: условно готов к production** при наличии базовой DevOps-обвязки проекта (CI, мониторинг, регресс-тесты на окружении Drupal).

Модуль демонстрирует хорошую инженерную основу для production-нагрузки:
- защита от redirect loop,
- запрет внешних URL для destination,
- кэширование компилированных правил,
- блокировка конкурентной пересборки кэша,
- диагностика «битых» правил,
- ограничение и очистка логов/статистики.

При этом остаются организационно-платформенные риски (вне бизнес-логики модуля): отсутствие зафиксированного CI-пайплайна и невозможность прогнать полноценные Drupal-тесты в текущем окружении.

---

## Что уже хорошо для production

1. **Безопасность destination и минимизация open redirect риска**
   - Валидация destination запрещает внешние URL через `UrlHelper::isExternal()`.
   - Runtime-нормализация destination выполняется через `path.validator`.

2. **Защита от redirect loop**
   - Сравнение текущего пути с destination.
   - Сравнение с internal path (`/node/{id}`, `/taxonomy/term/{id}`) и alias.
   - Дополнительная проверка по route name + route params.

3. **Производительность и устойчивость под нагрузкой**
   - Скомпилированный runtime-кэш правил в выделенном cache bin.
   - Lock на пересборку правил (`lock->acquire`) для защиты от «thundering herd».
   - In-request статический кэш нормализации destination.

4. **Эксплуатационная поддержка**
   - Логи и статистика вынесены в БД-таблицы.
   - Есть throttling одинаковых ошибок (`repeat_count`, `last_seen`).
   - Есть cron-очистка логов и статистики по лимитам.

5. **Административная безопасность**
   - Маршруты закрыты granular-permissions (view/manage/delete/logs/stats/settings).
   - Есть отдельная restricted permission для полного администрирования.

6. **Качество миграций/обновлений**
   - Набор update hooks для backfill конфигов, создания/эволюции таблиц и инвалидации кэша.

## Риски и ограничения

### High

1. **Нет подтвержденного CI/CD в репозитории**
   - Не обнаружены `composer.json`, `phpunit.xml*`, `.github/workflows` или аналогичные pipeline-конфиги в корне.
   - Следствие: нет гарантированного автоматического quality gate перед релизом.

2. **Нет возможности выполнить интеграционные Drupal-тесты в текущем окружении**
   - Тесты в `tests/src/...` присутствуют, но локально `vendor/bin/phpunit` отсутствует.
   - Следствие: текущий аудит не подтверждает runtime-поведение на целевой сборке Drupal.

### Medium

3. **Нет явного `hook_uninstall()`**
   - Таблицы и конфиг модуля могут оставаться после удаления модуля (зависит от политики проекта).
   - Для production-платформ обычно желательно явно зафиксировать стратегию cleanup.

4. **Диагностика и логирование завязаны на cron cleanup**
   - При неработающем cron объем данных будет расти быстрее ожидаемого.

### Low

5. **Смешение языков в интерфейсных сообщениях**
   - В коде встречаются строки на UA/EN; это не блокер production, но влияет на консистентность UX/локализации.

## Рекомендации перед production-выкаткой

### Blocker checklist (обязательно)

1. Подключить CI-пайплайн минимум с шагами:
   - `php -l` по модулю,
   - `phpunit` (kernel + functional),
   - статический анализ (минимум phpstan на согласованном уровне).

2. Запустить тесты модуля на целевом Drupal-стенде (версии PHP/Drupal как в production) и сохранить артефакты отчета.

3. Проверить и зафиксировать cron SLA (частота, alerting при сбоях), т.к. на нем держится housekeeping логов/статистики.

### Желательно

4. Добавить `hook_uninstall()` (или документировать осознанное сохранение таблиц/данных).
5. Добавить runbook инцидентов:
   - что делать при всплеске redirect loops,
   - как отключать конфликтные правила,
   - как безопасно чистить накопленные логи.
6. Стандартизировать локализации UI-строк.

## Выполненные проверки в рамках аудита

- Синтаксическая проверка PHP-файлов модуля (`php -l`) — ошибок не обнаружено.
- Поиск явных debug-антипаттернов (`TODO/FIXME/var_dump/dump/die`) — совпадений не обнаружено.
- Обзор критичных участков:
  - event subscriber,
  - matcher,
  - failure logger,
  - маршруты/permissions,
  - install/update hooks.

---

## Заключение

С точки зрения **архитектуры и прикладной логики** модуль реализован на хорошем production-уровне.

С точки зрения **release-процесса и операционной надежности** для финального «Go Live» необходимо закрыть инфраструктурные пункты (CI + обязательный тест-прогон на Drupal-стенде + контролируемый cron).
